## Summary

Phase 2 is a pure Sanity data layer phase: define document schemas in TypeScript, register them in `sanity.config.ts`, write typed GROQ queries using `defineQuery`, and run `sanity typegen generate` to produce `sanity.types.ts`. The project already has Sanity v4 (`sanity@^4.22.0`), `next-sanity@^11.6.12`, `@sanity/image-url@^1.2.0`, and scaffolded helpers (`client.ts`, `image.ts`, `live.ts`) from Phase 1 — no new packages are needed.

The two core schema documents are `product` (the dispensary product with media gallery) and `siteUser` (for the auth gate in Phase 3). For the media gallery requirement (PROD-04), the pattern is a single array field whose members can be either `image` type or `file` type — Sanity natively supports mixed-type arrays. Image URLs are resolved at query time using the `->` dereference operator; file (video) URLs work the same way. TypeScript types are generated automatically by running `npx sanity typegen generate` after writing queries with `defineQuery` from the `groq` package.

The project already has `src/sanity/structure.ts` with `S.documentTypeListItems()` which auto-discovers registered schema types — no Structure Builder customization is needed for this phase. The only required wiring is adding schema type definitions to `src/sanity/schemaTypes/index.ts` and updating `sanity.config.ts` to import them.

**Primary recommendation:** Define schemas with `defineType`/`defineField`, use a single `media` array with mixed image/file members for PROD-04, write all GROQ queries using `defineQuery` from `groq`, run `npx sanity typegen generate`, and include `sanity.types.ts` in `tsconfig.json`. Do not add any new npm packages.

### Pattern 1: Document Schema Definition

**What:** Define each Sanity document type in its own file using `defineType` and `defineField`.
**When to use:** Every document type that appears in Studio.

```typescript
// Source: https://www.sanity.io/docs/studio/schema-types
// src/sanity/schemaTypes/product.ts
import { defineType, defineField, defineArrayMember } from 'sanity'

export const productType = defineType({
  name: 'product',
  title: 'Product',
  type: 'document',
  fields: [
    defineField({
      name: 'name',
      title: 'Name',
      type: 'string',
      validation: (Rule) => Rule.required(),
    }),
    defineField({
      name: 'strainType',
      title: 'Strain Type',
      type: 'string',
      options: {
        list: [
          { title: 'Sativa', value: 'sativa' },
          { title: 'Hybrid', value: 'hybrid' },
          { title: 'Indica', value: 'indica' },
        ],
        layout: 'radio',
      },
      validation: (Rule) => Rule.required(),
    }),
    defineField({
      name: 'price',
      title: 'Price (USD)',
      type: 'number',
      validation: (Rule) => Rule.required().min(0),
    }),
    defineField({
      name: 'thcPercent',
      title: 'THC %',
      type: 'number',
      validation: (Rule) => Rule.required().min(0).max(100),
    }),
    defineField({
      name: 'cbdPercent',
      title: 'CBD %',
      type: 'number',
      validation: (Rule) => Rule.required().min(0).max(100),
    }),
    defineField({
      name: 'description',
      title: 'Description',
      type: 'text',
    }),
    defineField({
      name: 'effects',
      title: 'Effects',
      type: 'array',
      of: [defineArrayMember({ type: 'string' })],
      description: 'e.g. Relaxing, Euphoric, Creative',
    }),
    defineField({
      name: 'media',
      title: 'Images & Videos',
      type: 'array',
      of: [
        defineArrayMember({
          type: 'image',
          title: 'Image',
          options: { hotspot: true },
          fields: [
            defineField({ name: 'alt', type: 'string', title: 'Alt text' }),
          ],
        }),
        defineArrayMember({
          type: 'file',
          title: 'Video',
          options: { accept: 'video/*' },
          fields: [
            defineField({ name: 'caption', type: 'string', title: 'Caption' }),
          ],
        }),
      ],
      validation: (Rule) => Rule.required().min(1),
    }),
  ],
  preview: {
    select: {
      title: 'name',
      subtitle: 'strainType',
      media: 'media.0',
    },
  },
})
```

### Pattern 3: Typed GROQ Queries with defineQuery

**What:** Write all queries in a single `queries.ts` file using `defineQuery` so TypeGen can analyze them.
**When to use:** Every data fetch call from the Next.js app.

```typescript
// Source: https://www.sanity.io/docs/apis-and-sdks/sanity-typegen
// src/sanity/lib/queries.ts
import { defineQuery } from 'groq'

// Fetch all products ordered by name — resolves image URL via asset->url
export const ALL_PRODUCTS_QUERY = defineQuery(`
  *[_type == "product"] | order(name asc) {
    _id,
    _type,
    name,
    strainType,
    price,
    thcPercent,
    cbdPercent,
    description,
    effects,
    "media": media[] {
      _type,
      _key,
      alt,
      caption,
      "url": asset->url,
      "mimeType": asset->mimeType
    }
  }
`)

// Fetch single product by _id
export const PRODUCT_BY_ID_QUERY = defineQuery(`
  *[_type == "product" && _id == $id][0] {
    _id,
    _type,
    name,
    strainType,
    price,
    thcPercent,
    cbdPercent,
    description,
    effects,
    "media": media[] {
      _type,
      _key,
      alt,
      caption,
      "url": asset->url,
      "mimeType": asset->mimeType
    }
  }
`)

// Fetch products by strain type — used for Phase 4 carousels
export const PRODUCTS_BY_STRAIN_QUERY = defineQuery(`
  *[_type == "product" && strainType == $strainType] | order(name asc) {
    _id,
    name,
    strainType,
    price,
    thcPercent,
    cbdPercent,
    "primaryImage": media[_type == "image"][0] {
      alt,
      "url": asset->url
    }
  }
`)
```

### Pattern 5: TypeGen Workflow

**What:** Generate TypeScript types from schemas and GROQ queries.

```bash
# 1. Extract schema to schema.json
npx sanity schema extract

# 2. Generate sanity.types.ts from schema + queries
npx sanity typegen generate
```

Then update `tsconfig.json`:
```json
{
  "include": ["src/**/*", "sanity.types.ts", "next-env.d.ts"]
}
```

Optionally enable auto-generation in `sanity.cli.ts`:
```typescript
// sanity.cli.ts
import { defineCliConfig } from 'sanity/cli'

export default defineCliConfig({
  api: { projectId: ..., dataset: ... },
  typegen: {
    enabled: true,
    path: './src/**/*.{ts,tsx}',
    generates: './sanity.types.ts',
    overloadClientMethods: true,
  },
})
```

### Anti-Patterns to Avoid

- **Storing raw asset references in queries without `->` dereference:** The query will return `{ _ref: "image-abc-100x100-jpg" }` — not a URL. Always use `asset->url` in projections.
- **Writing inline queries instead of `defineQuery`:** TypeGen cannot detect `client.fetch(groq\`...\`)` inline — always assign to a named variable with `defineQuery(...)`.
- **Not adding `sanity.types.ts` to `tsconfig.json` includes:** TypeScript will not see the generated types and will infer `any`.
- **Using `type: 'image'` for videos:** Image type lacks the `accept` filtering and correct asset type. Use `type: 'file'` with `options: { accept: 'video/*' }` for video uploads.
- **Putting the `effects` field as a single `string` with comma separation:** Use `array` of strings — better for Studio UX, GROQ filtering, and future display as chips.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Image URL construction from asset references | Custom URL builder from projectId + assetId | `@sanity/image-url` `urlFor(source).url()` | Handles CDN, transformations, hotspot cropping, format conversion |
| TypeScript types for Sanity documents | Manual interface definitions | `npx sanity typegen generate` | Stays in sync with schema automatically; manual types drift |
| Mixed image/video array in Studio | Custom asset type or plugin | Native Sanity `array` with `of: [image, file]` | Sanity Studio natively supports mixed-type arrays with type selector |
| GROQ query result types | `as Product[]` type casts | `defineQuery` + TypeGen | Auto-infers exact shape including optional fields and nested objects |
| Image asset URL resolution | Manual dereference code | GROQ `asset->url` projection | Resolved at the Sanity API level, no runtime overhead |

**Key insight:** Sanity's tooling (TypeGen, image-url builder, native array types) handles every hard part of this phase. The implementation is purely declarative schema definitions + GROQ queries.

## Common Pitfalls

### Pitfall 1: Image vs. File Type for Videos

**What goes wrong:** Developer uses `type: 'image'` for video uploads — Studio shows the wrong asset picker (image-specific), and `@sanity/image-url` fails on video assets.
**Why it happens:** The requirement says "images/videos" and image type is the first that comes to mind.
**How to avoid:** Use `type: 'file'` with `options: { accept: 'video/*' }` for video members in the media array. Use `type: 'image'` only for actual image members.
**Warning signs:** GROQ query returns `_type: "sanity.imageAsset"` for video items; `urlFor()` throws on video refs.

### Pitfall 2: Asset References Not Dereferenced in Queries

**What goes wrong:** GROQ query returns `{ asset: { _ref: "image-...", _type: "reference" } }` instead of a URL. Success criterion 3 (resolving image URLs) fails.
**Why it happens:** Query written as `media[]` without the `{ "url": asset->url }` projection.
**How to avoid:** Every query that needs a URL must use `asset->url` with the dereference operator `->`. Test by running the query in the Sanity Vision tool (`/studio`) before writing TypeScript code.
**Warning signs:** Frontend receives objects with `_ref` instead of URL strings.

### Pitfall 3: TypeGen Not Detecting Queries

**What goes wrong:** `sanity typegen generate` produces `sanity.types.ts` with no GROQ query types — `client.fetch()` returns `any`.
**Why it happens:** Queries written inline (`client.fetch(groq\`...\`)`) instead of as named `defineQuery` variables. TypeGen only detects queries assigned to variables.
**How to avoid:** Always write `export const MY_QUERY = defineQuery(\`...\`)` in the queries file. Never inline the query string directly in `client.fetch()`.
**Warning signs:** `sanity.types.ts` exists but contains only schema types, no `*Result` query types.

### Pitfall 4: schema.json Out of Sync

**What goes wrong:** TypeGen generates types from a stale `schema.json`, leading to type/schema divergence. The project compiles but types are wrong.
**Why it happens:** Developer adds a field to the schema but forgets to re-run `npx sanity schema extract` before `typegen generate`.
**How to avoid:** Always run `schema extract` before `typegen generate`. Consider enabling `typegen.enabled: true` in `sanity.cli.ts` to automate this.
**Warning signs:** TypeScript type does not include a field that exists in the schema.

### Pitfall 5: Validation Rules Only Enforced in Studio

**What goes wrong:** API mutations (e.g., programmatic writes in Phase 3 auth flow) bypass schema validation, allowing required fields to be empty.
**Why it happens:** Sanity validation is client-side within Studio — it does not enforce at the Content Lake API level.
**How to avoid:** For the auth Phase 3 siteUser document (which will be created programmatically), validate required fields in application code before calling `client.create()`. Schema validation in Studio is for UX guidance only.
**Warning signs:** A document exists in Sanity without required fields populated.

### Pitfall 6: `useCdn: true` on Client Used for Mutations

**What goes wrong:** Studio or server-side writes using CDN client fail silently or return stale data.
**Why it happens:** The existing `client.ts` has `useCdn: true` — fine for reads, but mutations must use a non-CDN client with a write-token.
**How to avoid:** For Phase 3 (when programmatic document creation is needed), create a separate write client with `useCdn: false` and the write-scoped API token. For Phase 2, the read client is sufficient — all writes go through Studio UI, not code.
**Warning signs:** `client.create()` fails with auth errors or writes don't appear.

## Open Questions

1. **passwordHash field on siteUser**
   - What we know: Phase 3 will use Auth.js v5 — it manages sessions. The `siteUser` document may only need `email + status`, not a password hash (Auth.js handles auth separately from Sanity).
   - What's unclear: Will Phase 3 store any credential in Sanity, or will Sanity only store `status` and `email` as identity references?
   - Recommendation: Define `siteUser` with `email`, `name`, and `status` only for Phase 2. Omit `passwordHash` — Phase 3 research will clarify the Auth.js/Sanity relationship. Do not add fields that may be removed.

2. **Media array order — image first vs. mixed**
   - What we know: PROD-04 says "at least one image/video." Success criterion 1 says "at least one image/video." The product preview in Studio uses `media.0`.
   - What's unclear: Should the schema enforce that the first media item is an image (for the product card primary image)?
   - Recommendation: Add a Studio hint in the `media` field description ("Upload images first — the first image appears as the product card thumbnail") but do not enforce it with validation. The GROQ query will use `media[_type == "image"][0]` for primary image selection regardless of position.

3. **Sanity API version**
   - What we know: Current `env.ts` uses `"2024-07-11"`. Sanity v4.22.0 is installed.
   - What's unclear: Whether a newer API version should be used for any v4-specific features.
   - Recommendation: Keep `"2024-07-11"` — it's stable and sufficient for all Phase 2 operations. Do not change without a specific reason.
